# Fitness-RAG
# Knowledge Model (v1.0)

---

# 1. Purpose

The Fitness-RAG platform is not simply an exercise database.

It is an AI-powered fitness planning system that combines structured exercise knowledge, scientific training guidelines, user constraints, and retrieval-augmented generation (RAG) to create safe, explainable, and personalized workout programs.

The purpose of this document is to define the conceptual knowledge model that governs the entire application.

This document is intentionally independent of:

- MongoDB
- Express
- FastAPI
- React
- Atlas Vector Search

It describes **what knowledge exists**, not **how it is stored**.

---

# 2. Core Philosophy

Every piece of information in the system belongs to exactly one of four categories.

```
Knowledge

├── Static Knowledge
├── Scientific Knowledge
├── User Knowledge
└── Generated Knowledge
```

---

# 3. Knowledge Categories

## 3.1 Static Knowledge

Facts that rarely change.

Examples:

- Exercises
- Muscles
- Equipment
- Categories
- Movement Patterns

These are objective truths.

Example:

```
Bench Press

Primary Muscle:
Chest

Equipment:
Barbell

Difficulty:
Intermediate
```

These are never generated by AI.

---

## 3.2 Scientific Knowledge

Evidence-based recommendations.

Examples

- ACSM Guidelines
- NSCA Position Stands
- Injury Contraindications
- Hypertrophy Principles
- Strength Programming

Unlike exercises, these are documents.

Example

```
Hypertrophy

Recommended weekly volume:

10–20 sets

Recommended reps:

6–20
```

---

## 3.3 User Knowledge

Information supplied by users.

Examples

Age

Experience

Goal

Equipment

Available Days

Injuries

Workout History

Preferences

---

## 3.4 Generated Knowledge

Created dynamically.

Examples

Weekly Workout

Daily Workout

Exercise Order

Rest Times

Exercise Explanations

These should never become part of the knowledge base.

The Fitness-RAG system contains the following conceptual entities.

---

Represents one physical exercise.

Responsible for:

- Name
- Instructions
- Images
- Equipment
- Difficulty
- Muscles
- Category

Not responsible for:

- Workout planning
- Weekly volume
- Scheduling

# 7. Retrieval Strategy

The RAG pipeline uses three different retrieval mechanisms.

## Structural Retrieval

Used for

Equipment

Difficulty

Muscles

Goals

Experience

Categories

Injuries

This uses MongoDB queries.

---

## Semantic Retrieval

Used for

Exercise Instructions

Programming Documents

Technique Notes

Injury Articles

Guidelines

This uses Atlas Vector Search.

---

## LLM Reasoning

Responsible for

Exercise Selection

Workout Split

Scheduling

Progression

Explanation Generation

JSON Formatting

The LLM must never invent exercises.

# 11. Design Principles

1. Facts are stored, never generated.
2. Scientific knowledge is retrieved, never memorized.
3. User constraints always override preferences.
4. Every recommendation must be traceable to retrieved evidence.
5. The LLM may only use retrieved exercises.
6. Generated workouts must pass validation before storage.
7. Structured data is filtered before semantic retrieval.
8. Every exercise recommendation should be explainable.

# Conclusion

The Fitness-RAG platform is fundamentally a knowledge-driven system.

Its architecture is built around well-defined domain entities rather than database tables. By separating static exercise facts, scientific guidance, user-specific information, and AI-generated outputs, the system remains modular, explainable, and scalable.

This knowledge model serves as the foundation for all subsequent phases, including dataset normalization, enrichment, MongoDB schema design, Atlas Vector Search, API development, and the RAG pipeline.
